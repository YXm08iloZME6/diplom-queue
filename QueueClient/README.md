# QueueClient — десктоп-клиент «Электронная очередь» (Avalonia + MVVM)

Один кроссплатформенный клиент на Avalonia UI 11 + CommunityToolkit.Mvvm, который
закрывает **все** роли веб-версии и работает в трёх режимах в зависимости от того,
на какой машине запущен:

| Режим (App.Mode) | Где ставится | Что показывает | Вход |
|---|---|---|---|
| `Login`  | обычный компьютер оператора/админа | экран входа → по роли рабочее место и/или админка | вручную (email/пароль) |
| `Kiosk`  | сенсорный терминал выдачи талонов | полноэкранный сценарий выдачи талона | авто, сервисной учёткой |
| `Display`| телевизор-табло в зале | активные вызовы + очередь, обновление вживую | авто, сервисной учёткой |

Клиент общается **только с REST API** (проект `Api`) по HTTP и слушает SignalR-хаб
`/queueHub` (события `NewTicket` / `UpdateTicket`). Своей БД у него нет.

---

## 1. Нужно ли встраивать в текущее решение?

Коротко: **отдельный проект, но можно положить в ту же солюшен `Queue.sln` для удобства.**

- Клиенту **нельзя** ссылаться на `Api` / `Web` / `Infrastructure` / `Domain` /
  `Application` как на проекты — они тянут EF Core, Npgsql, серверные зависимости.
  Десктоп должен оставаться лёгким и собираться без БД. Поэтому модели и enum'ы
  у клиента **свои** (тонкие копии в `Models/`).
- Удобно: добавить `QueueClient.csproj` в `Queue.sln` (одна солюшен, общие конфигурации
  запуска в Rider). Связи между проектами при этом нет — только сетевой вызов API.
- Если репозиторий должен быть «чистым клиентом» — можно держать `QueueClient`
  и отдельным репозиторием. На сборку это не влияет.

Добавить в существующую солюшен (по желанию):
```bash
# из папки с Queue.sln
dotnet sln Queue.sln add ../QueueClient/QueueClient.csproj
```

> На сервере, скорее всего, **ничего менять не нужно**. Один нюанс — авторизация хаба:
> сейчас контроллеры под `[Authorize]`, а хаб открыт. Клиент всё равно передаёт JWT в
> хаб (на будущее). Если позже повесишь `[Authorize]` на `QueueHub` — клиент уже готов.

## 2. Подготовка окружения (Mac + Rider)

```bash
dotnet --info          # нужен .NET 8 SDK
```
Шаблоны Avalonia для этого проекта не обязательны (проект уже готов). Пакеты подтянутся
сами при первом `dotnet restore` / открытии в Rider:
Avalonia 11, CommunityToolkit.Mvvm, Microsoft.Extensions.DependencyInjection,
Microsoft.AspNetCore.SignalR.Client.

Плагин **AvaloniaRider** в Rider — для превью `.axaml`.

---

## 3. Настройка `appsettings.json`

```json
{
  "App": {
    "ApiBaseUrl": "https://localhost:7148",
    "Mode": "Login",
    "ServiceAccountLogin": "kiosk@queue.local",
    "ServiceAccountPassword": "changeMe123",
    "IdleTimeoutSeconds": 60,
    "TicketDisplaySeconds": 10
  }
}
```

- `ApiBaseUrl` — адрес твоего API (в dev по умолчанию `https://localhost:7148`,
  см. `Api/Properties/launchSettings.json`).
- `Mode` — задаётся **на каждой машине свой**: `Login` на компьютерах сотрудников,
  `Kiosk` на терминале, `Display` на табло.
- `ServiceAccountLogin/Password` — учётка для авто-входа терминала и табло (вручную там
  логиниться некому). Заведи её заранее в админке (роль `operator` достаточно).

---

## 4. Учётные записи

- **Оператор / админ** (`Mode=Login`): входят сами по email/паролю. После входа клиент
  смотрит роли из JWT: `admin` видит все разделы (включая рабочее место), `operator` —
  только рабочее место и табло.
- **Терминал и табло** (`Mode=Kiosk`/`Display`): входят автоматически сервисной учёткой
  из конфига. Создай такого пользователя в админке (раздел «Сотрудники»).

---

## 5. Запуск на разработке

1. Подними API (он же поднимет SignalR-хаб `/queueHub`).
2. В `appsettings.json` укажи `ApiBaseUrl` и нужный `Mode`.
3. `dotnet run` (или Run в Rider).

В DEBUG включён обход самоподписанного HTTPS-сертификата localhost (и для HTTP-запросов,
и для SignalR). В проде убери — поставь нормальный сертификат на API.

Проверить все режимы на одной машине: меняй `Mode` и перезапускай. Для оператора
понадобится пользователь с ролью `operator` и назначенным окном (иначе панель попросит
начать смену).

## 6. Развёртывание на «железо»

Сборка self-contained (без установленного .NET на целевой машине):
```bash
# Терминал/ПК оператора на Windows
dotnet publish -c Release -r win-x64 --self-contained
# Табло/ПК на Linux
dotnet publish -c Release -r linux-x64 --self-contained
```
На каждую машину кладёшь свой `appsettings.json` с нужным `Mode` (и адресом API).

Подсказки по эксплуатации:
- **Терминал/табло** запускаются на весь экран без рамок автоматически (режимы Kiosk/Display).
  Автостарт при загрузке ОС настраивается средствами ОС (Автозагрузка/systemd/Task Scheduler).
- **Компьютер оператора** — обычное окно, режим `Login`.
- Сеть: десктоп-клиенту CORS не нужен (CORS — про браузеры). Достаточно, чтобы машина
  видела `ApiBaseUrl` и порт хаба.

## 8. Что стоит подогнать под свой бэк

- **Простой режим киоска**: `SimpleTicket` шлёт `ServiceId = Guid.Empty` + букву из
  настройки. Если `TicketService.CreateAsync` так не умеет — поправь логику в
  `KioskViewModel.SimpleTicket`.
- **Часовой пояс в статистике**: сервер возвращает даты как есть; форматирование — стандартное.
- **HTTPS в проде**: убрать обход сертификата (блоки `#if DEBUG`).
- Если повесишь `[Authorize]` на `QueueHub` — клиент уже передаёт токен, доп. правки не нужны.
